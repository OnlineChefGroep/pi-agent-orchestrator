# 2026-07-12 - UI String Truncation Bottleneck Optimization

**Systemic Bottleneck:** String truncation for UI formatting (`truncateToWidth`, `takeVisible`, `fastTruncate`, `padAndTruncate`) was allocating memory inside O(N) loops. The loops would continually append characters `out += text[i]`, resulting in quadratic string allocation overhead during hot-rendering loops for tables and agent dashboard widgets.
**Refactor Strategy:** Replaced character-by-character concatenation with a single-pass loop that counts visible width (skipping ANSI sequences) to find cut indices, then extracts the result with `text.slice()`. Output strings from `slice()`, padding, and ellipsis still allocate; the win is avoiding intermediate per-character string growth.
**Key Metric Shift:** Fewer intermediate allocations on the TUI hot path reduce GC pressure during dashboard renders. Verify with `npm run bench:all` (dashboard-render limits remain the acceptance gate).
**Actionable Principle:** For string truncating or ANSI processing in hot-rendering UI loops, prefer single-pass visible-length tallies that produce boundary indices for `slice()`, instead of building the result with `str += char` or array spreads.
